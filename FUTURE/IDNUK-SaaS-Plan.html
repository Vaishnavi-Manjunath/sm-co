Executive Summary

The short answer to your four questions

1 · Should we rewrite everything?

No A rewrite is the #1 way startups die. Your PHP + React app already works in production. We make it tenant-aware with a targeted refactor, not a rebuild.

2 · Should we change hosting?

Yes Shared cPanel cannot do programmatic provisioning, autoscaling or a 99%+ SLA. Move to managed cloud (VPS + managed MySQL + Cloudflare).

3 · What multi-tenancy model?

Database-per-tenant. Each customer gets their own copy of your current schema. Your data model is reused verbatim — strongest isolation, easiest sell.

4 · How long & how much?

~13–16 weeks to general availability. Infra for 200 tenants ≈ $300–500/mo (~₹25k–42k). Gross margin 80%+ at ₹999/tenant.

The rest of this deck details the architecture, hosting, security, costing, readiness gaps and a phase-by-phase rollout.

Recommended Architecture · Option B

Database-per-tenant, one control plane

Login resolves the tenant in the control plane → the app opens that tenant's database for the request. A query can never cross tenants, because the connection itself is scoped.

Your Key Constraint

How this leaves your data model untouched

Today the app talks to one fixed database:

define('DB_NAME', 'smanwefl_idnuk');

We change one thing — resolve the DB name from the logged-in tenant instead of a constant:

$db = connectTenant($auth->tenant_db);
// every existing query runs unchanged
  • Same tables, columns, views, indexes — a clone of today's schema
  • Your pilot (Sri Murugan & Co) simply becomes Tenant #1
  • migrateOnce() already self-migrates each DB on first hit after a deploy → schema changes roll out to all tenants automatically

What is genuinely new (small & separate)

  • A control-plane database: tenants, subscriptions, trials, plans, feature_flags, platform_users
  • A thin tenant-router in the auth layer (login → tenant → DB handle)
  • A provisioning job (create DB, import schema, seed defaults)
  • A billing module (Razorpay) + a marketing/signup site

None of this edits your business tables. It wraps around them.

"Customization for each customer"

Configurable, not forked — the golden SaaS rule

Every customer can look and behave differently — driven by configuration, never by separate copies of the code. One codebase serves all 200+ tenants.

Branding

  • Logo, shop name (EN/Tamil)
  • Print/bill templates
  • Receipt headers, terms
  • (already in your app)

Modules / Features

  • Per-tenant feature flags: Yard, Market, Reports, Tally export…
  • Plans unlock feature sets
  • Sidebar shows only what's enabled

Business rules

  • Commission %, coolie, sakku defaults
  • Units, rounding, languages
  • Working-date / day-lock policy

Discipline that keeps you sane at scale

If a customer needs something truly unique, it becomes a configurable option for everyone, not a private code branch. Per-customer code forks would make 200 customers impossible to maintain or update. Everything ships from one pipeline; customization is data.

Scale & Performance

Comfortable at 200, headroom for thousands

Why 200 tenants is easy here

  • A mandi is low-throughput (tens–hundreds of bills/day each). 200 of them is still light DB traffic.
  • MySQL handles thousands of schemas on one managed instance — we watch table-cache & connections, add ProxySQL only if needed.
  • Stateless app nodes + sessions in Redis → add nodes horizontally behind a load balancer in minutes.
  • Cloudflare CDN serves the React app globally; API stays light.

Performance budget (the "premium feel")

First app load< 2.0 s
Page/API response (p95)< 400 ms
Signup → ready< 30 s
Concurrent tenants200 → 2,000 w/o redesign

We already added prefetch + skeletons + sliding sessions; the SaaS tier adds CDN + Redis + connection pooling to hold these numbers under load.

Security & Compliance

Built for handling other people's money

Tenant isolation

  • DB-per-tenant = hard boundary
  • Tenant resolved from the signed session, never from client input
  • Per-tenant DB credentials

Platform security

  • TLS everywhere · Cloudflare WAF + rate limits
  • Secrets in a manager, not in git
  • Brute-force lockout (have it), 2FA for admins
  • Encrypted backups + restore drills

Compliance (India)

  • DPDP Act 2023: consent, data-retention, deletion on request
  • SaaS 18% GST invoicing
  • Audit log (have it) per tenant
  • Pen test before GA

Immediate pre-launch must-dos (carried from ops hardening)

Rotate the DB password currently in config/database.php and move all secrets out of the repo · force HTTPS · enable centralized encrypted backups with tested restores · set up uptime + error alerting.
